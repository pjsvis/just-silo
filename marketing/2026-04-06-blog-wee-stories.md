# The Wee-Story Principle

*Or: How I learned to stop worrying and trust the story*

---

## The Problem With Documentation

Technical docs are full of contradictions. We write:

> "The filesystem is the interface."
> 
> "Access via HTTP API."
> 
> "Agents read files."
> 
> "Agents poll endpoints."

These statements aren't wrong individually. But together? Chaos. Readers sense the incoherence even if they can't name it.

**The solution: Write the wee-story.**

---

## What Is A Wee-Story?

Awee-story is a 2-3 sentence narrative that captures the essential truth of a system.

Rules:
1. If you can't write it in 2-3 sentences, you don't understand it
2. If the wee-story contradicts itself, the docs have a contradiction
3. Wee-stories are semantic compression: big ideas → small stories

---

## Example: The Grain Silo

**Without wee-story:**
> "Define your data schema. Create a harvest recipe. Set thresholds. Alert on breach."

**With wee-story:**
> "A farmer wants to know when grain is too wet. She creates a silo. The silo tells her."

The second version exposes what the first obscures:
- Who is the actor? (Farmer)
- What do they want? (To know)
- What's the mechanism? (Silo)

---

## Example: The Agent Drop-In

**Without wee-story:**
> "Agents discover silo via filesystem. They read .silo manifest. They execute just recipes."

**With wee-story:**
> "Agent enters directory. Reads rules. Does job. Done. Next agent inherits context via filesystem."

Notice what's missing: HTTP servers, API endpoints, polling mechanisms.

**Contradiction exposed:** If we say "filesystem is the interface," we can't also say "use the SSE endpoint." Both cannot be primary.

**Resolution:** The filesystem IS the interface. HTTP is a convenience layer. This is stated in the docs but contradicted by the existence of `silo-api-server.ts`.

---

## The Semantic Compression Test

Before publishing any doc, run the test:

```
Can you distill this into a wee-story?

If yes: The concept is clear.
If no: There's a contradiction hiding.
```

---

## Real Example: Data Stratification

**Initial doc claim:**
> "Silo files are divided into scaffold and throughput."

**Wee-story attempt:**
> "Scaffold files are the contract. Throughput files are the stuff flowing through. Never commit throughput."

**Exposed contradiction:** The README shows `harvest.jsonl` as example data, implying it's part of the structure. But `.gitignore` excludes it. Which is it?

**Resolution:** 
- `harvest.jsonl` is *input* — provided by user, not generated
- `data.jsonl` is *throughput* — generated by pipeline
- Update README to distinguish input vs throughput

---

## The Wee-Story Review

When reviewing PRs, ask:

1. **Does the code match the wee-story?**
2. **Does the docs match the wee-story?**
3. **Do the wee-stories contradict each other?**

If the answer to any is "no," there's work to do.

---

## The Wee-Story Library

We maintain a library of wee-stories in `marketing/wee-stories.md`. Before writing new docs, add to the library. Before reviewing, check the library.

The library is:
- A compression tool: Big docs → small stories
- A contradiction detector: If you can't write it, it's inconsistent
- A review aid: "Does this PR respect the wee-stories?"
