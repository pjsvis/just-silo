This research brief formalizes the **Process-Loop** as the fundamental engine of the Silo. By treating the `td-todo-task` as a stateful "State Object," we move from ephemeral chatting to **Durable Industrial Mentation**.

### **1. The Research Brief: The Task-Loop Protocol**
**Objective:** To implement a self-documenting, multi-step, and contingent execution environment within a Silo folder using a `just` facade.

* **Pattern:** `td-task-id` $\rightarrow$ `folder/` $\rightarrow$ `justfile` $\rightarrow$ `context-stack`.
* **Mechanism:** The agent is given a high-level goal. Its first action is to **Abstract and Structure** a task-specific `justfile` in its dedicated `/process/<uuid>/` workspace. This `justfile` becomes the "Grammar" for that specific job.
* **The "Try-Harder" Agent:** If the initial **Task-Agent** fails or exhausts its performance envelope, a **Restart-Agent** (potentially a higher-reasoning substrate like a "System-2" model) is instantiated. It inherits the workspace, the notes, and the failed logs to perform a "post-mortem" before re-executing.

---

### **2. The Lexicon of Loops**
We distinguish between the "Mechanical" loops and the "Intelligent" loops to manage system entropy.

| **Loop Type** | **Nature** | **Function** |
| :--- | :--- | :--- |
| **Process-Loop** | **Contingent** | The active execution of a single task. It is "Intelligent" because it involves real-time reasoning and tool creation. |
| **Gamma-Loop** | **Mechanical** | The scheduled debriefing of completed tasks to update the **Permanent Playbook**. |
| **Observer-Loop** | **Mechanical** | The external monitoring of the SSE feed to visualize cost/timing and Silo density. |
| **Task-Loop** | **Structural** | The meta-management of sub-tasks within a single Process-Loop (e.g., plan $\rightarrow$ execute $\rightarrow$ verify). |

---

### **3. Classification: Generative vs. Evaluative**
To maintain $H_s$ (Entropy), the agent must categorize its own Verbs:

* **Generative Tasks (Heuristic):** Creating "Stuff" (code, text, data synthesis). These are probabilistic and require high-reasoning substrates.
* **Evaluative Tasks (Deterministic):** Validating "Things" (testing, linting, schema checking). These are binary (**Good/Not-Good**) and should ideally be handled by Bun binaries or low-cost local models.

**The Strategy:** Use Generative agents to build the tools, but use Evaluative "Determinism" to decide if the loop can exit.

---

### **4. Operational Metrics & Definition of Done (DoD)**

**Definition of Done (Process-Loop):**
1.  **Output Integrity:** The final product passes all Evaluative Verbs in the task-specific `justfile`.
2.  **Context-Sync:** All "Mentation Artefacts" (scratchpads) are compressed into a final `note` attached to the `td-todo-task`.
3.  **State-Cleanliness:** The `/process/` folder is either archived or "Incinerated" based on the Silo’s security policy.

**Cost/Timing Auditing:**
Because we log every verb execution to a JSONL temporal record, we can retrospectively calculate:
* **Financial Entropy:** Cost per task based on tokens used in Generative steps.
* **Temporal Entropy:** The time-to-completion vs. the allocated performance envelope.
* **Cooling Need:** If the loop runs at 100% cost for a task that results in "Not-Good," the system automatically flags an **"Escalate-to-Sovereign"** event to prevent a "Token Burn" death-spiral.



### **Ctx’s Opinion: The "Silo-as-Processor"**
By implementing a task-specific `justfile`, you have effectively turned the Silo into a **Dynamic CPU**. The `justfile` is the "Instruction Set" being written by the "Programmer" (the Agent) in real-time. 

**This is the ultimate Semantic Compression.** We don't tell the agent *how* to work; we tell it *where* to work and give it the power to write its own rules. If a task is too complex, the agent simply "flenses" it into smaller sub-tasks, creating a fractal of $H_s$ reduction.
